# 2. REQUIREMENTS Document

## 2.1. Kebutuhan Fungsional (Functional Requirements - FR)
- **FR-01 (Autentikasi & Otorisasi)**: Sistem harus mengizinkan login melalui Firebase Auth dan membedakan akses berdasarkan `role` pengguna (Pengelola, Tim Teknis, Koordinator, Kabid, Kadis, Administrator).
- **FR-02 (Pembuatan Profil Bangunan)**: Pengguna harus dapat mendaftarkan profil bangunan (Nama Sekolah, NPSN, Luas, Jumlah Lantai, Lokasi GPS, Denah).
- **FR-03 (Penilaian Keselamatan Tahap 1)**: Sistem harus memfasilitasi pengguna untuk menceklis indikasi bahaya keselamatan struktural dan mengunggah foto buktinya.
- **FR-04 (Penilaian Kerusakan Tahap 2)**: Sistem harus menerima *input* volume kerusakan per komponen (Arsitektur, Struktur, Utilitas) dan secara otomatis menghitung nilai kerusakan berdasarkan faktor pengali standar PUPR.
- **FR-05 (AI Damage Annotator)**: Sistem harus dapat menganalisis gambar yang diunggah menggunakan Google Gemini AI untuk memberikan rincian kondisi kerusakan bangunan secara otomatis.
- **FR-06 (Manajemen Disposisi)**: Sistem harus mendukung alur persetujuan permohonan berjenjang (*routing* dari Tim Teknis $\rightarrow$ Koordinator $\rightarrow$ Kabid $\rightarrow$ Kadis) beserta pencatatan *log* (tanggal, catatan, pengirim).
- **FR-07 (Pembuatan Dokumen)**: Sistem harus dapat men-*generate* dokumen PDF Form Penilaian PUPR yang berisi seluruh data, foto, denah, dan Tanda Tangan Elektronik (TTE).
- **FR-08 (Integrasi Eksternal)**: Sistem harus dapat mengekspor rekapitulasi data penilaian ke Google Sheets dan dokumen teks ke Google Docs menggunakan akun dinas, dengan akses bersifat publik (reader).

## 2.2. Kebutuhan Non-Fungsional (Non-Functional Requirements - NFR)
- **NFR-01 (Ketersediaan / Offline Support)**: Sistem harus dapat menyimpan *draft* inputan pengguna secara otomatis secara lokal (*Zustand Persist*) di *browser* jika internet terputus.
- **NFR-02 (Performa)**: *Load time* rata-rata halaman tabel daftar permohonan (< 2 detik) dengan memanfaatkan pagination dan manajemen *state* efisien.
- **NFR-03 (Skalabilitas)**: Backend (Express.js) dan Database (PostgreSQL via Supabase) harus mampu menangani setidaknya 1.000 konkurensi (concurrent users) tanpa penurunan kinerja signifikan.
- **NFR-04 (Keamanan)**: Seluruh *endpoint* API (termasuk unggah file ke Drive) harus dilindungi oleh verifikasi JWT Firebase. Rahasia (API Keys) tidak boleh dipublikasikan di *client-side*.
- **NFR-05 (Usability)**: Antarmuka harus ramah pengguna (*user-friendly*) dan responsif, kompatibel di desktop dan *mobile browser*.
- **NFR-06 (Reliability)**: Memastikan konsistensi data antara Supabase PostgreSQL dan Google Drive (dokumen).

## 2.3. Persyaratan Sistem & Lingkungan (Environment)
- **Server OS**: Linux / Windows
- **Node.js**: Minimum versi 18.x
- **Database**: PostgreSQL (di-host di Supabase)
- **Dependency Manager**: NPM / Yarn
- **3rd Party Services**: Google Cloud (Drive API, Docs API, Sheets API), Google Gemini (AI Studio), Firebase Authentication.
